# Этапы 0–1: подготовка и разграничение слоёв

## Цели
- **Скорость**: минимизировать избыточные операции в UI‑потоке.
- **Читаемость**: явно разделить ответственность между UI, ViewModel и сервисами.
- **Тестируемость**: обеспечить изоляцию логики в сервисах/VM.
- **Снижение связности**: UI не зависит от деталей работы с настройками и API.

## Критерии готовности (чек‑лист)
- [ ] UI‑код содержит только события и привязки.
- [ ] ViewModel хранит состояние и команды.
- [ ] Сервисный слой — единая точка доступа к бизнес‑операциям.
- [ ] Отсутствуют прямые обращения к `AppSettings` из UI‑кода.
- [ ] Ключевые потоки подключения/синхронизации покрыты сервисами.

## Границы изменений
- **Не трогаем** UI/UX (разметка, стили, визуальные компоненты).
- **Не меняем** публичные API.
- **Не меняем** форматы данных.

## План задач для этапа 1
1. Централизовать получение учётных данных PlayCanvas в сервисе.
2. Удалить прямую работу с `AppSettings` из code‑behind, использовать сервис.
3. Подготовить ViewModel к самостоятельной работе с учётными данными.

## Факт выполнения (текущий шаг)
- Создан сервис для получения учётных данных и дешифрации API‑ключа.
- Code‑behind использует сервис вместо прямого чтения настроек.
- MainViewModel инициализирует Username/ApiKey через сервис.
